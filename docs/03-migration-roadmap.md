code# 重做实施路线图

## 1. 总原则

这次重做不要追求“一次把所有功能全部改完”，而是要按主链路推进。

最稳的顺序是：

1. 先把新版骨架搭起来
2. 先打通认证和博客
3. 再迁移打卡
4. 再迁移社交和聊天
5. 最后替换旧项目

## 2. 推荐阶段划分

### 阶段 0：冻结旧项目，做资产盘点

目标：

- 把旧项目当成参考系统，不再继续大改
- 明确哪些能力要迁移，哪些暂时放弃

建议输出：

- 模块清单
- 旧接口清单
- 旧数据结构清单
- 页面清单

完成标准：

- 旧项目功能边界清楚
- 新项目范围清楚

### 阶段 1：搭新版工程骨架

目标：

- 新建新版目录
- 跑通前后端开发环境
- 接入基础数据库

建议先完成：

- `apps/web`
- `apps/api`
- `packages/shared`
- `content/posts`
- `uploads`
- `docs`

完成标准：

- 前端能启动
- 后端能启动
- 数据库能连通
- 有基础健康检查接口

### 阶段 2：先打通认证系统

一期只做一种登录方式，建议优先邮箱。

需要实现：

- 注册
- 登录
- 退出
- 会话恢复
- 用户资料读取
- 用户资料更新

数据库侧需要先落地：

- `users`
- `user_profiles`
- `user_identities`
- `sessions`

完成标准：

- 用户可注册并登录
- 刷新页面后仍能恢复会话
- 可以在个人中心修改资料

### 阶段 3：重做博客模块

这是最值得最先迁移的业务模块，因为它既能验证 Markdown，也能验证上传和权限。

需要实现：

- 博客列表
- 博客详情
- 发布文章
- 保存 Markdown 文件
- 封面图或正文图片上传
- 评论
- 点赞

这一步的关键不是“编辑器做多漂亮”，而是把内容链路做对：

1. 输入 Markdown
2. 存成 `.md`
3. 元数据入库
4. 详情页正确渲染

完成标准：

- 能发布一篇真实 Markdown 文章
- 能正确渲染标题、列表、代码块、图片
- 评论和点赞走数据库

### 阶段 4：迁移打卡模块

需要实现：

- 新建打卡项
- 完成记录
- 历史统计
- 可见性控制
- 好友提醒

数据库建议落地：

- `habits`
- `habit_records`
- `habit_reminders`

完成标准：

- 用户能创建并完成打卡项
- 好友可见的打卡项能被读取和提醒

### 阶段 5：迁移社交和聊天

这是系统里关系最复杂的一块，建议放到博客和打卡之后。

需要实现：

- 好友申请
- 好友关系
- 会话列表
- 私聊消息
- 未读计数
- 通知联动

数据库建议落地：

- `friend_requests`
- `friendships`
- `conversations`
- `messages`
- `message_reads`
- `notifications`

完成标准：

- 申请好友、通过好友、聊天这条链路能跑通
- 未读和通知正确更新

### 阶段 6：补充高级功能

这阶段再考虑：

- 手机登录
- 微信登录
- QQ 登录
- 草稿箱
- 搜索
- 标签页
- 后台管理
- 对象存储
- 邮件通知

## 3. 你应该先放弃哪些“诱人但不该一期做”的东西

为了让重做真的落地，一期先不要做这些：

- 同时接四种登录方式
- 超复杂富文本编辑器
- 实时聊天 WebSocket
- 后台管理系统
- 搜索系统
- 云对象存储
- 消息推送

这些都能做，但不应该卡在第一阶段。

## 4. 一期最小可上线版本

如果目标是尽快有个“新版可用版本”，我建议你的一期范围只保留：

- 邮箱登录
- 个人资料
- 博客发布
- Markdown 渲染
- 图片上传
- 评论点赞
- 基础打卡

先不要做：

- 微信
- QQ
- 聊天
- 番茄钟
- 复杂通知中心

## 5. 旧数据怎么迁移

当前旧数据在 `data/store.json`，可以考虑写一次性迁移脚本。

建议迁移顺序：

1. 用户
2. 资料
3. 博客元数据
4. Markdown 内容文件
5. 评论
6. 点赞
7. 打卡项和记录
8. 好友关系
9. 消息
10. 通知

其中博客部分建议这样迁移：

- 从旧 `blogs[].content` 取出正文
- 为每篇文章生成一个 `.md` 文件
- front matter 写入标题、作者、时间、可见性等元数据
- 数据库里记录文件路径和索引信息

## 6. 推荐的开发节奏

不要用“大重构”思维，要用“小阶段交付”思维。

建议节奏：

1. 一个阶段只完成一条业务闭环
2. 每完成一阶段就手测
3. 每完成一阶段就提交 Git
4. 阶段之间不要交叉返工

## 7. 我建议你的第一步

如果你准备正式开工，第一步不是改旧项目，而是：

1. 在同级目录新建一个新版工程目录
2. 搭起 `web + api + db + content` 骨架
3. 先把“邮箱登录 + 博客发布 + Markdown 文件保存”打通

为什么是这条链路：

- 它能同时验证认证
- 能验证数据库
- 能验证文件存储
- 能验证真正的 Markdown

这条链路打通以后，后面的打卡和社交都会顺很多。

## 8. 一句话路线图

先搭骨架，再通认证，再做博客，再迁打卡，最后接社交和聊天。不要一开始就做全功能和全登录方式。
