# 当前项目诊断

## 1. 结论

这个项目适合重做，不适合继续在现有结构上直接加功能。

原因不是“代码多”本身，而是**代码边界已经塌陷**：

- 前端所有页面状态和渲染都集中在一个文件里
- 后端所有路由、校验、存储、上传、通知逻辑都集中在一个文件里
- 数据仍然存放在本地 JSON 文件，不适合继续扩展
- Markdown 是手写解析器，不是真正的标准实现
- 认证模型只支持当前这套“选用户 + 密码”的单一流程，无法平滑扩展到手机、邮箱、微信、QQ

所以最合理的做法是：

1. 保留旧项目作为参考实现
2. 新建新版工程
3. 按模块逐步迁移博客、打卡、聊天、通知等能力

## 2. 我看到的当前结构

当前项目核心文件非常集中：

- `server.js`：2143 行
- `public/app.js`：5037 行
- `public/styles.css`：2320 行

总计大约 9500 行核心代码。

同时，数据仍落在 `data/store.json`，结构包括：

- `users`
- `sessions`
- `blogs`
- `checkinItems`
- `notifications`
- `friendRequests`
- `messages`

这说明当前实际上已经不是一个“纯前端静态页”，而是一个**前后端都写在一起的单体原型**。

## 3. 当前已经实现的业务能力

从接口和前端状态可以看出，项目已经覆盖这些模块：

- 认证：注册、登录、找回密码、会话恢复、退出、删除用户
- 个人资料：头像、性别、简介、地区
- 博客：发布、详情、评论、点赞、删除
- 打卡：创建项目、完成记录、好友提醒
- 社交：用户主页、好友申请、好友关系
- 聊天：会话列表、消息收发、已读状态
- 通知：点赞、评论、好友申请、聊天提醒
- 工具：番茄钟
- 上传：头像上传、博客图片上传

这也是为什么我不建议继续在旧代码里硬拆。因为功能已经不少，但边界没有沉淀出来。

## 4. 当前最核心的问题

### 4.1 前端是“大状态 + 大渲染 + 大事件分发”

`public/app.js` 里一个全局 `state` 挂了几乎全部模块：

- 认证
- 博客
- 打卡
- 聊天
- 通知
- 资料
- 番茄钟
- 弹窗
- 上传

这会直接导致下面的问题：

- 任意模块变更都容易影响整体渲染
- 很难做独立测试
- 复用能力很差
- 后续接入路由和权限会越来越乱

### 4.2 后端是“大 if 路由”

`server.js` 目前通过一串 `if (pathname === "...")` 处理所有接口。

这样做在早期开发很快，但到你现在这个规模就会出现明显问题：

- 认证、博客、聊天、通知逻辑完全混杂
- 参数校验分散
- 权限判断分散
- 数据写入分散
- 后面接数据库时会变成“大迁移手术”

### 4.3 数据存储还是文件 JSON

现在的 `data/store.json` 对演示可以，对产品化不行。

主要问题：

- 没有事务
- 并发写入风险高
- 结构演进困难
- 查询能力很弱
- 难以支撑复杂关系

这个项目已经有以下关系型数据特征：

- 用户和多种身份绑定
- 用户和会话
- 用户和博客
- 博客和评论
- 博客和点赞
- 用户之间的好友关系
- 私聊消息
- 通知事件
- 打卡项目和完成记录

这些都已经明显超出 JSON 文件最适合承担的边界。

### 4.4 认证模型太窄

当前注册依赖：

- 用户名
- 密码
- 密保问题
- 密保答案
- 头像

登录依赖：

- 先从已有用户列表里选中用户
- 再输入密码

这个模型不适合扩展成你想要的多登录方式。因为你后面真正需要的是：

- 一个内部用户表
- 一个身份绑定表
- 一套验证码或 OAuth 回调流程
- 一套会话和刷新机制

### 4.5 Markdown 不是“真正的 Markdown”

现在的 Markdown 是前端自己写的解析函数，虽然已经支持一部分语法，但本质还是自定义规则。

这类实现后面会长期踩坑：

- 语法兼容性不稳定
- 扩展功能难加
- 代码块高亮不完整
- 表格和嵌套列表边界容易错
- 安全过滤容易出问题

这部分应该直接换成成熟解析库，不要继续手写。

### 4.6 存在重复函数定义，说明代码已经开始“复制式增长”

我在 `public/app.js` 里确认到了重复函数名：

- `renderBlogModule`
- `renderBlogPost`
- `renderBlogDetailModal`
- `renderUserSpaceModal`

这通常意味着：

- 中后期改功能时出现整段复制
- 局部重构没有真正收束
- 维护成本会继续指数上升

这也是支持“新建项目重做”的直接证据。

## 5. 哪些东西值得保留

虽然建议重做，但旧项目并不是废的。下面这些内容值得继续利用：

### 5.1 业务边界可以保留

你已经验证过这些模块确实有价值：

- 博客
- 打卡
- 社交
- 聊天
- 通知
- 个人主页

新版可以继续保留这几块，只是要重建结构。

### 5.2 数据模型思路可以保留

比如这些概念在新版里依然成立：

- 用户
- 会话
- 博客
- 评论
- 点赞
- 好友申请
- 消息
- 通知
- 打卡记录

只是要把它们从 JSON 迁移到数据库。

### 5.3 上传目录思路可以保留

头像和博客图片分目录存放，这个思路是对的。新版可以继续保留，但需要引入文件元数据表。

## 6. 为什么我建议“先新建，再迁移”

因为你现在面临的是**架构重建**，不是“局部修补”。

如果继续在旧项目里做下面这些事：

- 引数据库
- 加多种登录方式
- 改真实 Markdown
- 拆前后端
- 抽模块

那么你会同时面对：

- 旧数据兼容
- 旧状态兼容
- 旧接口兼容
- 旧 DOM 渲染兼容

这会让复杂度叠在一起。

更稳的方式是：

1. 先设计新版边界
2. 先把认证和博客打通
3. 再逐步迁移打卡、社交、聊天、通知

## 7. 一句话建议

旧项目继续保留作为参考样板；新版项目单独起目录，用数据库和模块化结构重建，不在原有 `server.js + public/app.js` 基础上继续大修。
